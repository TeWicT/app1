# Изменения после версии, отражённой в probrep.tex (до текущего состояния)

## Полный обзор
Ниже приведено подробное описание всех доработок, появившихся после версии отчёта `probrep.tex`. Изменения затрагивают модели и права, middleware, представления (включая вспомогательные обработчики), шаблоны, статику, работу с legacy‑файлом `index.clj`, а также импорт исторических данных. Особое внимание уделено корректной визуализации статусов файлов, надёжной синхронизации `index.clj` и роли администратора с «Админ‑панелью».

## Модели данных
1) AdminProfile (новая модель)
   - Назначение: выдаёт доступ к странице «Админ‑панель».
   - Поля: `user` (OneToOne к пользователю), `full_name`, `created_at`.
   - Поведение: наличие связанной записи `AdminProfile` у пользователя интерпретируется как административная роль. Проставляется исключительно через стандартную Django‑админку (LDAP не задействован).

2) Topic
   - Добавлен флаг `is_active` — состояние видимости темы для студентов.
   - Инвариант: тему с заявками в статусах `pending` или `approved` скрывать нельзя.
   - При обновлении/удалении темы инициируется синхронизация `index.clj` у затронутых студентов (см. ниже).

3) Enrollment
   - При выборе темы у преподавателя без кафедры поле `department` сохраняется пустым значением (в интерфейсе это отображается как «Кафедра: Не указана»).
   - При удалении темы у преподавателя данные о теме в `Enrollment` очищаются (заголовок, ФИО руководителя и др.), чтобы не оставлять «висячих» ссылок и рассинхронизированных данных.

## Middleware
1) Расширение FoundYearMiddleware
   - В объект запроса добавляется `request.admin_profile` (по аналогии со `student_profile` и `teacher_profile`).
   - Благодаря этому в шаблонах и представлениях можно корректно проверять права и показывать/скрывать элементы интерфейса.

2) Исправления, влияющие на навигацию
   - Для пользователей без студентского профиля больше не формируются ссылки, требующие `foundyear` (исправлена ошибка `NoReverseMatch` при пустом годе у «чистого» администратора).
   - Счётчики уведомлений в шапке остаются в рабочем состоянии: преподавателям показывается число «ожидающих заявок», студентам — уведомления по темам (реализация в middleware осталась совместимой).

## Аутентификация и LDAP
1) `webd_core/auth_backends.py` (кастомный backend)
   - Добавлен/расширен backend `LdapBackend`, подключённый в `AUTHENTICATION_BACKENDS`.
   - Логика входа:
     - поиск пользователя по `uid` в LDAP;
     - bind от имени найденного DN с введённым паролем;
     - при успехе — создание/обновление локального `auth_user`.
   - Автосинхронизация профилей по DN:
     - для `ou=students`: создаются/актуализируются `Student`, `Group` (для текущего года) и `Enrollment`;
     - для `ou=faculty`: создаётся/актуализируется `TeacherProfile`.
   - Важный инвариант: административная роль не берётся из LDAP и по-прежнему задаётся только через `AdminProfile`.

2) Параметры LDAP в `app1/settings.py`
   - Вынесены настройки `LDAP_SERVER`, `LDAP_PORT`, `LDAP_USE_SSL`, `LDAP_BASE_DN` в переменные окружения.
   - Сохранён fallback на стандартный backend Django (`ModelBackend`) в цепочке backend'ов.

## Представления (views) и вспомогательная логика
1) Страница «Поиск и просмотр работ» (`page_webd`)
   - Список лет отсортирован по убыванию (максимальный — сверху). Пункт «Все» — всегда внизу списка.
   - Список групп формируется по всем доступным годам (ранее — только по текущему).
   - Для тем с пустой кафедрой в выводе отображается «Кафедра: Не указана».
   - В таблице результатов статусы файлов и цветовая индикация рассчитываются не по дате загрузки в БД, а строго по `index.clj :creation-time` (подробности в разделе «Работа с index.clj»).

2) Страница «Админ‑панель» (`admin_panel_view`)
   - Доступ: только пользователи с `request.admin_profile`.
   - Содержимое:
     - Сводный список преподавателей и их тем.
     - Действия администратора:
       - `create_topic` — создание темы с привязкой к выбранному преподавателю;
       - `update_topic` — редактирование полей темы (название, описание, курс, вместимость, направление и т.д.);
       - `delete_topic` — удаление темы с каскадной очисткой связанных данных в `Enrollment` (если студенты были привязаны) и синхронизацией `index.clj`;
       - `toggle_active` — скрыть/показать тему (с проверкой отсутствия заявок `pending/approved`).
     - Результаты действий возвращаются в тот же шаблон, UI сходен со страницей преподавателя, чтобы минимизировать когнитивные переключения.
   - Вспомогательные обработчики:
     - `_handle_topic_update` — помимо обновления темы, приводит `Enrollment.department` к пустому значению, если у преподавателя нет кафедры; инициирует `_write_legacy_index_for_enrollment(...)`.
     - `_handle_topic_delete` — очищает данные в `Enrollment`, отвязывает студентов (если были), инициирует перезапись `index.clj`.

3) Страница «Предложить тему» (преподаватель)
   - Поддержаны действия:
     - `create_batch` (ранее существовало);
     - `update_topic` (inline‑редактирование);
     - `delete_topic` — полное удаление темы с очисткой данных у привязанных студентов и перезаписью `index.clj`;
     - `decision` — обработка заявок студентов (approve/reject);
     - `toggle_active` — скрыть/показать тему.
   - Инварианты:
     - Скрывать тему, на которой есть заявки `pending/approved`, нельзя — кнопка скрытия отключена, добавлено пояснение в tooltip.
     - При `approve`/`reject` синхронизируются связанные данные в `Enrollment`. При `approve` кафедра ставится пустой строкой, если у преподавателя она не задана (UI — «Не указана»).

4) Страница «Выбрать тему» (студент)
   - Корректная работа фильтров по кафедрам с учётом состояния «Не указана».
   - Для тем преподавателей без кафедры выводится «Кафедра: Не указана».
   - Логика заявок сохранена: нельзя иметь несколько ожидающих (pending) одновременно.

5) «Регистрация работы» (`page_identity`) и «Загрузка файлов» (`upload_view`)
   - После сохранения формы регистрации и после каждой операции загрузки/удаления файла вызывается `_write_legacy_index_for_enrollment(...)`, чтобы `index.clj` всегда отражал текущее состояние.

6) Построение строк таблицы результатов (`_build_row`)
   - Переработан расчёт «просрочено/в срок»:
     - Источник истины — только `index.clj :creation-time` конкретного файла.
     - Дедлайны вычисляются от календарного года начала работы:
       - Для «Пр. отчет» (`interim_report`) и «Пр. ЭП» (`interim_presentation`) — просрочка позже 01.01 следующего года.
       - Для остальных файлов — просрочка позже 01.06 следующего года.
     - Если `index.clj`/`creation-time` отсутствуют или неразборчивы — файл считается «в срок» (чтобы не красить ложноположительно).
   - Парсер `index.clj` расширен:
     - поддержаны форматы времени без и с микросекундами (`...Z` и `....ffffffZ`);
     - более толерантная обработка пробелов/дополнительных полей;
     - при необходимости выполняется точечный поиск временной метки для ключа `int-report` (если общий разбор не нашёл).
   - Для отображения в шаблоне теперь передаются:
     - `in_time` (bool) — признак сдачи в срок (определяет CSS‑класс ячейки);
     - `deadline` (строка) — человеческое представление дедлайна для tooltip;
     - `uploaded_at` (строка) — фактическое время загрузки из `index.clj` для tooltip.

7) Перезапись `index.clj` (`_write_legacy_index_for_enrollment`)
   - Восстановлена корректность форматирования (убран лишний разделительный символ).
   - Вызывается из всех мест, где меняется состояние, отображаемое в legacy‑структуре: правки темы, решение по заявке, регистрация работы, загрузка/удаление файла, удаление темы.

8) Формирование PDF (`webd_core/utils/pdf.py`)
   - Логика рендера PDF через WeasyPrint переведена на «ленивый импорт».
   - Это устраняет падение импорта модуля на окружениях без системных зависимостей WeasyPrint (типичный случай локальной Windows-разработки).
   - При недоступности WeasyPrint выбрасывается явная `RuntimeError` с пояснением, что генерацию PDF нужно выполнять в Docker-окружении.

## Работа с legacy index.clj
1) Источник времени загрузки
   - Отказались от использования `Document.uploaded_at` для вычисления просрочки, так как это приводило к ложноположительной просрочке при импорте старых файлов.
   - Единственный источник истины — поле `:creation-time` соответствующего документа в `index.clj`.

2) Разбор форматов времени
   - Поддержаны варианты с микросекундами и без.
   - Разбор стал устойчивее к пробелам, переносам строк, дополнительным полям.
   - Добавлен точечный поиск для ключа `int-report`, если общий парсер его пропустил.

3) Дедлайны
   - На основе «календарного года начала работы» вычисляются «контрольные даты» следующего года:
     - для `interim_report`/`interim_presentation`: 01 января;
     - для остальных типов: 01 июня.
   - Сравнение ведётся в UTC (с учётом корректной замены на `datetime.timezone.utc` вместо неподдерживаемого `timezone.utc`).

4) Поведение при отсутствии данных
   - Если `index.clj` отсутствует или время не удалось из него извлечь, ячейка считается «в срок» и остаётся белой (чтобы не подсвечивать ошибочно).

## Исправления ошибок и устойчивость
1) `ModuleNotFoundError: No module named 'channels'`
   - Сообщение фиксировалось при запуске в некоторых окружениях. Дано пошаговое указание по установке зависимостей через `pip install -r requirements.txt` или точечно `channels daphne channels-redis` в виртуальном окружении (если используются фичи в проекте/окружении).

2) `NoReverseMatch` в шапке у чистого администратора
   - Исправлено условное отображение ссылок: student‑специфичные ссылки не строятся без `student_profile`, что устранило вызов `reverse('page_identity', {'foundyear': ''})`.

3) Ошибка `timezone.utc`
   - Заменено на `datetime.timezone.utc` для совместимости с используемой версией Django.

4) Красные ячейки у пустых/старых файлов
   - Перенастроены стили и логика вычисления «просрочки»: пустые — нейтральные, старые — определяются только по `index.clj`. Сняты ложноположительные подсветки.

## Инварианты и правила доменной логики
1) Скрытие темы
   - Недопустимо, если есть хотя бы одна заявка в статусах `pending` или `approved`.
   - Кнопка в UI отключена и сопровождается понятным tooltip.

2) Удаление темы
   - Очищает привязанные данные в `Enrollment` (название темы, руководитель, кафедра и т.д.).
   - Инициирует перезапись `index.clj` для затронутых студентов.

3) Кафедра «Не указана»
   - Если у преподавателя кафедра пустая, то в `Enrollment.department` сохраняется пустая строка.
   - На страницах: «Кафедра: Не указана».

## Ключевые сценарии, которые теперь поддерживаются
- Администратор:
  - Открывает «Админ‑панель», видит всех преподавателей и их темы.
  - Создаёт, редактирует, удаляет темы от имени преподавателей.
  - Скрывает/показывает темы (с защитой от скрытия занятых/ожидающих).
  - Любые изменения немедленно отражаются в `index.clj`.
- Преподаватель:
  - Скрывает/показывает собственные темы; не может скрыть при наличии заявок/принятых студентов.
  - Удаляет темы, при этом Enrollment соответствующих студентов очищается, `index.clj` синхронизируется.
- Студент:
  - Видит корректные кафедры (включая «Не указана»), выбирает темы.
  - На странице «Поиск и просмотр работ» видит реальные статусы просрочки по `index.clj`, всплывающие подсказки с дедлайнами и временем загрузки.
- Система:
  - Поддерживает отображение групп за все годы, годы сортируются по убыванию.
  - Поддерживает устойчивый парсинг `index.clj` в разных форматах времени.

### Описание таблиц

1) Year (Учебный год)
- Поля: `year` (уникальный), PK.
- Связи: `Group.year`, `Enrollment.year`.

2) Group (Группа)
- Поля: `name`, `is_latest`, FK `year`.
- Ограничения: уникальность (`name`, `year`).
- Связи: `Enrollment.group` (PROTECT).

3) Student (Студент)
- Поля: `login` (уникальный), `full_name`.
- Связи: `Enrollment.student`.

4) Enrollment (Запись обучения)
- Поля: `student` (FK), `year` (FK), `group` (FK), `courses`, тема/руководитель/должность/кафедра и др.
- Инварианты: при удалении темы соответствующие данные очищаются; кафедра может быть пустой (UI — «Не указана»).
- Связи: `Document.enrollment`, `TopicRequest.enrollment`.

5) Document (Документ)
- Поля: `enrollment` (FK), `doc_type`, `file`, `uploaded_at`.
- Семантика просрочки: определяется только по `index.clj :creation-time`, не по `uploaded_at`.

6) TeacherProfile (Профиль преподавателя)
- Поля: `user` (OneToOne к Django `auth_user`), `full_name`, `department` (может быть пустой).
- Связи: `Topic.teacher`.

7) Topic (Тема)
- Поля: `teacher` (FK), `title`, `description`, `department` (может быть пустой), `direction`, `course`, `capacity`, `is_active`.
- Инварианты: скрытие запрещено при `pending/approved` заявках.
- Действия: `create/update/delete/toggle_active` (в т.ч. из «Админ‑панели»).

8) TopicRequest (Заявка)
- Поля: `topic` (FK), `enrollment` (FK), `status` (`pending|approved|rejected`), `comment`, `decided_at`.
- Ограничения: уникальность (`topic`, `enrollment`).

9) AdminProfile (Профиль администратора)
- Поля: `user` (OneToOne к Django `auth_user`), `full_name`, `created_at`.
- Назначение: роль администратора (доступ к «Админ‑панели», операции с темами всех преподавателей).

10) DiscussionThread (Обсуждение темы)
- Поля: `topic` (FK на `Topic`), `created_at`.
- Назначение: контейнер сообщений по конкретной теме.

11) DiscussionMessage (Сообщение обсуждения)
- Поля: `thread` (FK), `author` (FK на `auth_user`), `text`, `created_at`.
- Назначение: хранение истории переписки; используется consumer’ом при доставке.

## Изменения в логике «просрочек» и tooltip
- Источник времени: `index.clj :creation-time` для конкретного документа.
- Допустимые форматы времени: с микросекундами и без.
- Дедлайны:
  - для `interim_report` и `interim_presentation`: позже 01.01 года N+1 — просрочка;
  - для остальных документов: позже 01.06 года N+1 — просрочка.
- Пустые ячейки не красим.
- Tooltip каждого PDF‑значка: «Дедлайн: …», «Загружено: …» (точные значения из `index.clj`).

## Обсуждения/чат в реальном времени (Django Channels)
Для поддержки обсуждений работ реализован чат с доставкой сообщений в реальном времени через WebSocket.

### Технологический стек
- Django Channels + ASGI (`daphne`) в качестве сервера.
- Redis (`channels-redis`) как channel layer в продакшене (включается через `CHANNEL_USE_REDIS=True`).
- Nginx умеет проксировать WebSocket (см. раздел «Инфраструктура»).

### Маршрутизация и инициализация
- ASGI‑приложение: в `app1/asgi.py` добавлен `ProtocolTypeRouter` с `AuthMiddlewareStack` и `URLRouter`.
- WebSocket‑маршрут: в `webd_core/routing.py` шаблон терпим к начальным/конечным слэшам: `^/?ws/discussion/(?P<thread_id>\d+)/?$`.
- URL‑маршруты страниц обсуждений: список обсуждений преподавателя и страница обсуждения подключены в `webd_core/urls.py`.

### Доступ и безопасность
- К подключению на `ws://.../ws/discussion/<thread_id>/` допускаются:
  - преподаватель — владелец темы обсуждения;
  - студент — только если у него утверждённая заявка (approved) на тему этого обсуждения.
- Доступ проверяется в consumer перед добавлением сокета в group‑канал.

### Consumer
- `webd_core/consumers.py` — `DiscussionConsumer` (async):
  - `connect`: верификация прав → `accept()` → присоединение к группе обсуждения;
  - `receive_json`: валидация, сохранение сообщения в БД, широковещательная отправка в group;
  - `discussion_message`: получение сообщения из group и отправка клиенту;
  - `disconnect`: выход из group.
- Сообщения сохраняются в БД сразу при приёме от клиента (надёжная история).

### Модели обсуждений
- `DiscussionThread`: ссылка на тему/участников, метаданные (время создания).
- `DiscussionMessage`: FK на `DiscussionThread`, автор (FK на `auth_user`), текст, время.

### Шаблоны и UI
- `page_discussion.html`: единая страница обсуждения
  - Заголовок: тема, руководитель, список участников;
  - Лента сообщений с автопрокруткой «к новому»;
  - Поле ввода, отправка через WebSocket без перезагрузки.
- `page_teacher_discussions.html`: список тем преподавателя с переходом в обсуждения.
- В `user_header.html` добавлены ссылки:
  - студент: «Обсуждение работы»;
  - преподаватель: «Обсуждение работ».

### Масштабирование и эксплуатация
- Redis channel layer обеспечивает горизонтальное масштабирование «web»‑сервиса.
- Nginx проксирует Upgrade/Connection.
- При деплое проверены:
  - множественные одновременные подключения к одному обсуждению;
  - авторизация и запрет доступа посторонним.

## Инфраструктура и деплой (Docker/Nginx)

### Dockerfile
- Приложение запускается как ASGI‑сервис через `daphne` (указание `app1.asgi:application`).
- Выполняется установка зависимостей (включая `channels`, `daphne`, при необходимости `channels-redis`).
- Собираются статики (`collectstatic`) в том же образе без dev‑инструментов.
- Параметры среды (например, `CSRF_TRUSTED_ORIGINS`, опции Redis) прокидываются на уровне контейнера.
- На старте контейнера выполняется импорт legacy-данных (`python manage.py import_legacy_projects`) перед запуском ASGI-сервера.

### docker-compose.yml
- Сервисы:
  - `db` (PostgreSQL 15): постоянный том `app1_pg_data`.
  - `web` (Django+ASGI): зависит от `db` и `redis`, публикует порт 8000 внутри сети, монтирует том `app1_static` и каталог `./media` для пользовательских файлов.
    - Важные переменные окружения:
      - `CHANNEL_USE_REDIS: "True"` — включает backend каналов на Redis;
      - `REDIS_HOST: "redis"`, `REDIS_PORT: "6379"`, `REDIS_DB: "0"` — параметры подключения channel layer.
  - `redis` (Redis 7‑alpine): постоянный том `app1_redis_data`, AOF включён.
  - `nginx` (1.27‑alpine): фронт HTTP, пробрасывает порт `80:80`, монтирует `nginx.conf`, `app1_static` (ro), `./media` (ro).
- Общие тома:
  - `app1_pg_data`, `app1_static`, `app1_redis_data`.

### Nginx
- Конфигурация учитывает заголовки Upgrade/Connection для корректной проксировки WebSocket трафика к ASGI‑приложению.
- Отдача статики и медиа производится напрямую из смонтированных томов/каталогов.

### Деплой на сервер
- Композиция развёрнута на сервере с использованием `docker compose up -d` после сборки образа `web`.
- На уровне окружения включён Redis‑channel layer (`CHANNEL_USE_REDIS=True`), что обеспечивает масштабируемые WebSocket‑подключения.
- Выполнены миграции, сборка статики и прогрев кэшей; проверены основные сценарии (админ‑панель, «Предложить тему», «Поиск и просмотр работ», загрузка/удаление файлов).

### Конфигурационные и служебные файлы
- `.dockerignore` уточнён: исключаются локальные БД/медиа/кеши/venv, чтобы не раздувать образ и не запекать runtime-данные в контейнер.
- `.gitignore` дополнен для IDE, локальной БД, медиа и `.env`.
- `env.example` обновлён актуальными переменными для запуска в контейнере (Django + PostgreSQL).
- `app1/settings.py` дополнен ASGI/Channels-конфигурацией, переключением channel layer (Redis/InMemory), параметрами `CSRF_TRUSTED_ORIGINS` и переменными для PostgreSQL/LDAP.
